# Data Model: Document Upload and Management

**Feature**: `001-document-upload-management`
**Date**: 2026-03-16

This document describes the planned data entities, relationships, and validation rules derived from the feature spec.

## Entity: Document

Represents a stored file plus its metadata.

### Fields

- `DocumentId` (int, PK)
- `Title` (string, required, max 255)
- `Description` (string?, optional, max 2000)
- `Category` (string, required, max 100)
  - Stores text values (per constraint):
    - `Project Documents`
    - `Team Resources`
    - `Personal Files`
    - `Reports`
    - `Presentations`
    - `Other`
- `Tags` (string?, optional, max 500)
  - Training-friendly approach: store as a delimited string (e.g., `"tag1;tag2"`).
- `UploaderUserId` (int, required, FK -> `User.UserId`)
- `ProjectId` (int?, optional, FK -> `Project.ProjectId`)
- `TaskId` (int?, optional, FK -> `TaskItem.TaskId`) *(for FR-017 task integration)*
- `FileNameOriginal` (string?, optional, max 255)
- `FileExtension` (string, required, max 20)
- `FileType` (string, required, max 255) *(MIME type)*
- `FileSizeBytes` (long, required)
- `StorageKey` (string, required, max 500)
  - Relative path/key generated by the system (e.g., `"{userId}/{projectOrPersonal}/{guid}{ext}"`).
- `UploadedUtc` (DateTime, required)
- `UpdatedUtc` (DateTime, required)
- `IsDeleted` (bool, required, default false)
- `DeletedUtc` (DateTime?, optional)

### Relationships

- `Document.Uploader` -> `User` (many documents per user)
- `Document.Project` -> `Project` (optional)
- `Document.Task` -> `TaskItem` (optional)
- `Document.Shares` -> `DocumentShare` (one-to-many)
- `Document.Activities` -> `DocumentActivity` (one-to-many)

### Validation Rules

- Title required
- Category required and must be one of the allowed values (enforced in service + UI)
- File size must be <= 25 MB (service + UI)
- File extension must match allowed whitelist (service)

### Indexes (EF)

- `(UploaderUserId, UploadedUtc DESC)` for "My Documents"
- `(ProjectId, UploadedUtc DESC)` for project documents
- `(TaskId, UploadedUtc DESC)` for task documents
- `Category`
- `IsDeleted` (if using soft delete queries frequently)

---

## Entity: DocumentShare

Represents explicit sharing from an owning context to a recipient.

### Fields

- `DocumentShareId` (int, PK)
- `DocumentId` (int, required, FK -> `Document.DocumentId`)
- `RecipientUserId` (int, required, FK -> `User.UserId`)
- `SharedByUserId` (int, required, FK -> `User.UserId`)
- `CreatedUtc` (DateTime, required)
- `Note` (string?, optional, max 500)

### Relationships

- Many shares per document
- Many shares per recipient

### Indexes

- `(RecipientUserId, CreatedUtc DESC)` for "Shared with me"
- Unique constraint candidate: `(DocumentId, RecipientUserId)` to prevent duplicate shares

---

## Entity: DocumentActivity

Audit log for document actions.

### Fields

- `DocumentActivityId` (int, PK)
- `DocumentId` (int, required, FK -> `Document.DocumentId`)
- `UserId` (int, required, FK -> `User.UserId`)
- `ActivityType` (string, required, max 50)
  - `Upload`, `Download`, `Preview`, `Delete`, `Share`, `MetadataUpdate`, `ReplaceFile`
- `CreatedUtc` (DateTime, required)
- `Details` (string?, optional, max 1000)

### Indexes

- `(DocumentId, CreatedUtc DESC)`
- `(UserId, CreatedUtc DESC)`
- `ActivityType`

---

## Notes on Soft Delete vs Hard Delete

- Spec requires permanent removal on delete confirmation.
- For training + audit requirements, we can either:
  - **Hard delete metadata + delete file** (simpler, but loses audit)
  - **Soft delete** (`IsDeleted`) + delete file and keep minimal audit record (recommended)

Planned approach: **Soft delete** documents and always record `DocumentActivity` for delete; the physical file is removed.

## EF Core Mapping Notes

- Add `DbSet<Document>`, `DbSet<DocumentShare>`, `DbSet<DocumentActivity>` to `ApplicationDbContext`
- Configure relationships + delete behaviors to avoid cascade pitfalls:
  - If user is deleted (not common in training), restrict or set null depending on entity.
  - Document deletion should clean up shares/activities depending on soft delete approach.
